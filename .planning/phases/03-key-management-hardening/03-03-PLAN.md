<objective>
Convert sync handler decrypt paths from getKeySync() to withKey() for key-before-content buffering, and add safety-net error display for genuine decryption failures.

Purpose: Implements the atomic key-before-content guarantee (KEYS-04 per D-05) -- encrypted content is held until the key is available, eliminating the #1 source of "[Content decryption failed]" errors. Adds visible error safety net (D-04) for the rare edge case where decryption genuinely fails.
Output: All sync handler decrypt paths use withKey() buffering. Safety-net error display in place.
</objective>

  <action>
    Per D-05: Buffer encrypted messages until the decryption key is confirmed available. Per D-03: Prevention, not error handling.

    **Classification step (do this first):** For each getKeySync() call in the 5 files, classify:
    - **(a) Must convert to withKey:** The null path leads to error handling, message drop, or "[Content decryption failed]". These are decrypt paths on incoming content.
    - **(b) Acceptable as-is:** The null path is handled gracefully (e.g., "show encrypted placeholder in sidebar", "skip optional metadata", "return early before encryption attempt"). These are render paths or optional enrichment.

    Document each classification as a code comment: `// KEYS-04: converted from getKeySync to withKey for key-before-content guarantee` or `// KEYS-04: getKeySync acceptable here -- null handled gracefully (shows placeholder)`

    **Conversion pattern for (a) sites:**

    Before (racy):
    ```typescript
    const chatKey = chatKeyManager.getKeySync(chatId);
    if (!chatKey) {
      console.error("[...] key not available");
      // error path, message lost
      return;
    }
    const decrypted = await decryptWithChatKey(encrypted, chatKey);
    ```

    After (buffered):
    ```typescript
    await chatKeyManager.withKey(chatId, "decrypt-{context}", async (chatKey) => {
      const decrypted = await decryptWithChatKey(encrypted, chatKey);
      // ... process decrypted content
    });
    ```

    **Important conversion rules:**
    1. The withKey callback must contain ALL code that depends on the key, not just the decrypt call. Move the post-decrypt processing inside the callback.
    2. If the function was synchronous before, it becomes async. Update callers if needed (most sync handlers are already async).
    3. Use descriptive labels: "decrypt-ai-title", "decrypt-broadcast-metadata", "decrypt-phased-sync-title" etc.
    4. For chatMetadataCache.ts: sidebar cache decrypt is a render path. If the key is not immediately available, showing "[Encrypted]" placeholder is acceptable. Classify as (b) but add an async re-render path using getKey() (not withKey -- sidebar should not block on queue).

    **For chatSyncServiceHandlersCoreSync.ts:** Lines 60-85 use `decryptChatKeyWithMasterKey` directly to unwrap, then decrypt content. This is a phased sync path. Convert the decrypt operations to use withKey after the unwrap establishes the key via `chatKeyManager.receiveKeyFromServer` or `chatKeyManager.injectKey`.

    **Anti-pattern to avoid (from RESEARCH.md):** Never use getKeySync() as the sole key acquisition in a decrypt path. If converting a site and the null path is an error, it MUST become withKey or getKey.

    Per D-04: After conversion, the remaining error paths (inside withKey callbacks, when decryptWithChatKey itself throws) should log `console.error("[ChatKeyManager] Decryption failed for chat ${chatId}:", error)`. These will be surfaced in UI by the safety-net task (Task 2).
