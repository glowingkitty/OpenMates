# Encryption Architecture

End-to-end encryption architecture for OpenMates after the Phase 1-4 rebuild.
All chat content is encrypted client-side before syncing to the server.

## Overview

[2-3 paragraphs: core value (every chat decrypts on every device), threat model (user encrypting their own data, not multi-party E2E), crypto primitives used (AES-GCM via Web Crypto API, NaCl via tweetnacl for key wrapping)]

## Module Map

[Mermaid diagram showing all encryption modules, their responsibilities, and dependencies]

## Data Flow: Encrypt -> Sync -> Decrypt

[Mermaid sequence diagram showing the full lifecycle of an encrypted message:]
1. User types message in tab
2. ChatKeyManager.withKey() provides the chat key
3. MessageEncryptor.encryptWithChatKey() encrypts the plaintext
4. chatSyncServiceSenders sends ciphertext over WebSocket
5. Server stores ciphertext (never sees plaintext)
6. Second device receives ciphertext via WebSocket sync
7. ChatKeyManager.withKey() provides the key (from IDB or BroadcastChannel)
8. MessageEncryptor.decryptWithChatKey() decrypts
9. UI renders plaintext

## Module Boundaries

### MessageEncryptor (encryption/MessageEncryptor.ts)
- Stateless: takes key + plaintext, returns ciphertext (and reverse)
- Handles OM-header format (2 magic bytes + 4 fingerprint bytes + IV + ciphertext)
- No state, no side effects, no IDB access
- Imported by sync handlers via `import { encryptWithChatKey } from '../encryption/MessageEncryptor'`

### MetadataEncryptor (encryption/MetadataEncryptor.ts)
- Stateless: handles title, embed metadata, and other non-message encrypted fields
- Master-key operations: encryptWithMasterKeyDirect, decryptWithMasterKeyDirect
- Embed-key operations: encryptEmbedKey, decryptEmbedKey
- No state, no side effects

### ChatKeyManager (encryption/ChatKeyManager.ts)
- The single authority for all key operations
- withKey(chatId, callback): provides key to callback, handles loading/caching
- Web Locks mutex: `om-chatkey-{chatId}` prevents duplicate key generation across tabs
- BroadcastChannel: propagates keyLoaded events across tabs (with loop prevention via _receivingFromBroadcast flag)
- State machine: unloaded -> loading -> ready, loading -> failed, retry paths
- Key-before-content guarantee: withKey() buffers operations until key is available

### cryptoService.ts (services/cryptoService.ts)
- Re-export barrel: preserves backwards compat for dynamic imports
- Re-exports encryptWithChatKey, decryptWithChatKey from MessageEncryptor
- Re-exports encryptWithMasterKeyDirect, decryptWithMasterKeyDirect from MetadataEncryptor
- No implementation code, pure re-exports

## Key Lifecycle

[Mermaid diagram showing key derivation and distribution:]
- User credential -> PBKDF2 -> master key (CryptoKey)
- Master key wraps/unwraps chat keys (AES-KW)
- Chat key: 32-byte AES-256-GCM symmetric key (one per chat)
- Cross-device: deterministic derivation from same credential = same master key
- Cross-tab: BroadcastChannel propagation + shared IndexedDB

## Cross-Device Key Distribution

[Brief explanation: deterministic derivation is the distribution mechanism. Same password on a new device derives the same master key, which unwraps the same chat keys from the server.]

## Cross-Tab Key Propagation

[Brief explanation of BroadcastChannel flow with the keyLoaded handler and pending-ops guard pattern from Phase 3]

## Sync Handler Architecture

[Mermaid diagram of sender module structure after Phase 4 split:]
- sendersChatMessages.ts — message send/edit/delete
- sendersChatManagement.ts — chat create/rename/archive
- sendersDrafts.ts — draft sync
- sendersEmbeds.ts — embed content sync
- sendersSync.ts — reconnection and background sync
- index.ts — barrel re-export

[Explain: all handlers import from MessageEncryptor/MetadataEncryptor directly, never from cryptoService. Enforced by import-audit.test.ts.]

## WebSocket Key Delivery

[Brief explanation of the ack protocol from Phase 4:]
- Sender device sends key via WebSocket
- Recipient device receives, injects into ChatKeyManager, sends key_received ack
- Server relays key_delivery_confirmed back to sender
- Ack is fire-and-forget: failure never blocks key injection
